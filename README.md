# Content Orchestrator

Phase 1 scaffold for the TakkenAI content operations platform.

## Features

- Topic management
- Weekly automated topic selection
- Distribution planning
- Content task generation
- Content artifact storage
- Publisher pull / publish-result APIs
- Publisher claim / lease flow
- SEO asset recording
- Alembic migration scaffold
- Unified JSON error responses

## Default database

By default the app uses SQLite in the local application data directory:

- Windows: `%LOCALAPPDATA%\ContentOrchestrator\content_orchestrator.sqlite3`

This keeps the database stable across restarts while avoiding file locking issues on synced or network-backed workspaces.

Set `DATABASE_URL` if you want a stable project-local or shared database.

When the schema changes, point `DATABASE_URL` to a fresh SQLite file for local testing or run Alembic migrations before restarting the app.

Override with environment variables:

- `DATABASE_URL`
- `APP_NAME`
- `APP_ENV`
- `PUBLISHER_API_KEY`
- `TOPIC_STRATEGY_PATH`
- `TOPIC_SIGNAL_SOURCES_PATH`
- `ENABLE_TOPIC_SELECTION_SCHEDULER`
- `TOPIC_SELECTION_CRON`
- `ENABLE_TOPIC_REFILL_SCHEDULER`
- `TOPIC_REFILL_CRON`
- `AUTOMATION_TIMEZONE`

Database resolution note:

- The app and Alembic now share the same database URL resolution logic.
- If `DATABASE_URL` is unset, both default to the local application data SQLite database.
- If `DATABASE_URL=sqlite:///./...`, both resolve it relative to the project root.
- For local development, prefer a project-local SQLite path such as `sqlite:///./data/content_orchestrator.sqlite3` so the app can write tasks, artifacts, and publish state reliably.

## Weekly automated topic selection

The project can now create and plan topics automatically once per week.

How it works:

1. Define topic direction in `data/topic_strategy.json`.
2. Define signal sources in `data/topic_signal_sources.json`.
3. The automation merges strategy keywords with signal keywords from manual seeds, JSON feeds, published artifact metadata, and SEO anchor feedback.
4. Candidates are scored for freshness, platform coverage, business goal weight, signal weight, and recent cluster saturation.
5. Top candidates are inserted into `topics` with `status=ready`.
6. Each created topic is immediately planned into distribution tasks.

Signal feed examples:

- `data/input/keyword_feed.json`: external collector output that can be generated by another script or crawler
- `GET /automation/topic-selection/signals`: inspect all currently collected signals and their scores
- `GET /automation/topic-selection/feedback`: inspect internal winner topics and platform coverage gaps derived from published results

Strategy and signal config roles:

- `data/topic_strategy.json`: decides topic clusters, business goals, title templates, platforms, and which signal sources each cluster should consume
- `data/topic_signal_sources.json`: decides where keyword signals come from and how strong each source should be

Internal feedback loop:

- `published_topic_winners`: finds keywords that already produced published artifacts across multiple platforms
- `platform_gap_feedback`: finds keywords that have some published platforms but still have unfulfilled target platforms
- `seo_anchor_feedback`: reuses anchor text from recorded SEO assets

Default schedule:

- Every Monday at `09:00` in `Asia/Shanghai`
- Cron expression: `0 9 * * 1`

Manual trigger:

```bash
curl -X POST "http://127.0.0.1:8000/automation/topic-selection/run"
```

Force rerun for the current week:

```bash
curl -X POST "http://127.0.0.1:8000/automation/topic-selection/run?force=true"
```

View recent runs:

```bash
curl "http://127.0.0.1:8000/automation/topic-selection/runs"
```

## Automatic topic inventory refill

The project can automatically supplement topic inventory when publishable topics are low.

How it works:

1. `TopicInventoryAlertService` counts pending topics by platform.
2. `TopicRefillService` compares pending counts against `data/topic_strategy.json` `auto_refill.threshold`.
3. Low-inventory platforms are replenished toward `auto_refill.target_pending`.
4. Candidates are built from strategy keywords, signal feeds, published feedback, and SEO anchors.
5. Candidates are scored for platform gap coverage, long-tail keyword value, SEO backlink fit, target URL coverage, signal strength, business goal weight, and priority.
6. Existing local topics and existing Feishu topic rows are normalized and used for duplicate prevention.
7. Created topics are written to the Feishu topic table when Feishu is configured, then inserted locally and planned into distribution tasks.

Default refill schedule:

- Every weekday at `16:45` in `Asia/Shanghai`
- Cron expression: `45 16 * * mon-fri`

Manual preview:

```bash
curl -X POST "http://127.0.0.1:8000/automation/topic-refill/run?dry_run=true"
```

Manual run:

```bash
curl -X POST "http://127.0.0.1:8000/automation/topic-refill/run"
```

Force rerun for the current day:

```bash
curl -X POST "http://127.0.0.1:8000/automation/topic-refill/run?force=true"
```

View recent refill runs:


## Publish Autopilot

The publish autopilot coordinates the currently integrated publish lanes:

- `note_a`
- `note_b`
- `ameba`
- `hatena_b`

It recovers stale `publishing` artifacts and stale `running` tasks, ensures each lane has a `publish_pending` artifact, triggers the publisher service, waits for the final artifact status, and records each run in `publishrun`.

Run all lanes:

```powershell
.\scripts\invoke-publish-autopilot.ps1
```

Run one lane:

```powershell
.\scripts\invoke-publish-autopilot.ps1 -Lane note_a
```

Dry run:

```powershell
.\scripts\invoke-publish-autopilot.ps1 -DryRun
```

API:

```bash
curl -X POST "http://127.0.0.1:8000/automation/publish-autopilot/run?lanes=note_a&wait=true"
curl "http://127.0.0.1:8000/automation/publish-autopilot/runs"
```

## Phase 2 Publisher Flow

Recommended publisher integration flow:

1. Publisher calls `POST /publisher/claims` with platform and consumer name.
2. Orchestrator atomically moves artifacts from `publish_pending` to `publishing`.
3. Publisher posts result to `POST /publisher/artifacts/{artifact_id}/publish-result`.

Publisher API auth:

- If `PUBLISHER_API_KEY` is set, publisher endpoints require `Authorization: Bearer <key>`.
- If `PUBLISHER_API_KEY` is unset, publisher endpoints remain open for local integration and testing.

Manual recovery for failed publishes:

- Re-queue a failed artifact with `POST /artifacts/{artifact_id}/requeue`.
- This resets the artifact back to `publish_pending` so a publisher worker can claim it again.
- Optional request body fields:
  `requested_by`, `reason`, `clear_error`

Example:

```bash
curl -X POST "http://127.0.0.1:8000/artifacts/{artifact_id}/requeue" ^
  -H "Content-Type: application/json" ^
  -d "{\"requested_by\": \"ops\", \"reason\": \"session fixed\", \"clear_error\": true}"
```

Sample worker:

```bash
python examples/note_publisher_worker.py --base-url http://127.0.0.1:8000
```

The sample worker is intentionally simple and shows where an existing `note-auto-publisher` can plug in its real login and publish logic.

Local publisher triggers:

```powershell
.\scripts\invoke-note-orch-publish.ps1 -Account note_a
.\scripts\invoke-note-orch-publish.ps1 -Account note_b
.\scripts\invoke-ameba-orch-publish.ps1
```

The Ameba trigger starts the existing `note-auto-publisher` Ameba service in Orchestrator mode, claims one `ameba` artifact, publishes it with the existing Ameba browser flow, and reports the result back to Content Orchestrator.

Switch Ameba automation mode:

```powershell
.\scripts\Set-AmebaPublisherMode.ps1 -Mode Orchestrator
.\scripts\Set-AmebaPublisherMode.ps1 -Mode Legacy
```

`Orchestrator` mode registers the Windows task `note-auto-publisher-ameba-publish` to call `invoke-ameba-orch-publish.ps1` every day at 14:30 local time. `Legacy` mode preserves the old Ameba path by registering the same task to start the existing 8011 Ameba service at 14:20 local time, letting its built-in scheduler handle publishing.
